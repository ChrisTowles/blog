I opened my `~/.claude/settings.json` last week and found exactly one hook. A notification handler. That's it. One hook, sitting there by itself like a single picture on an otherwise empty wall.

I'd configured it months ago and never thought about hooks again. Meanwhile, I've been writing CLAUDE.md files, setting up agent teams, configuring plugins, building custom skills -- doing everything _around_ hooks while ignoring the hooks themselves.

Then Boris Cherny's thread hit my timeline and I felt that specific flavor of embarrassment where you realize you've been doing something the hard way for months. Boris created Claude Code. When the guy who built the thing shares how he actually uses it, and half his tips involve a feature you've barely touched -- yeah, that stings a little.

## One Hook, One Wall

Here's my entire hooks setup before this week:

```json
{
  "hooks": {
    "Notification": [
      {
        "matcher": "",
        "hooks": [
          {
            "type": "command",
            "command": "bun run /home/ctowles/code/p/dotfiles/config/claude/notify.ts"
          }
        ]
      }
    ]
  }
}
```

One hook. Notifications. That's the Claude Code equivalent of buying a sports car and only driving it to the mailbox.

## The Distinction That Clicked

The thing Boris said that rewired my thinking was deceptively simple. CLAUDE.md says "please do this." A hook says "this _will_ happen, period." One is a suggestion to a language model. The other is code that executes at a precise lifecycle moment.

I'd been relying entirely on suggestions. And for the most part, they worked. But "most of the time" isn't good enough when you're running five parallel sessions and walking away from the keyboard. I've come back to find Claude forgot a convention after context compaction. I've caught a session trying to force-push to main. Every time, I thought "I should add a guardrail for that" and then didn't.

Hooks are the guardrail.

## The Events I'd Been Ignoring

Claude Code fires hooks at every major lifecycle moment. The [official docs](https://code.claude.com/docs/en/hooks) cover them all, but five changed how I think about the tool. Not because they're the most powerful -- because they solve problems I was already having.

**SessionStart** -- Every new Claude session, I used to paste recent git history into the conversation. Every. Single. Time. Copy the last five commits, paste them in, add a note about what I was working on. It was the kind of ritual you don't question until someone shows you the SessionStart hook and you realize a three-line script can compute that context fresh every time Claude boots up.

**PreToolUse** -- Last month, one of my parallel Claude sessions tried to force-push to main. The permissions system caught it, but I still felt my stomach drop. That's the kind of near-miss that makes you want a second layer of defense -- something that fires before every tool execution and can block dangerous commands with a regex, no matter what the permissions config says.

**PostToolUse** -- Every time Claude writes or edits a file, this hook runs your formatter automatically. I was opening PRs and finding formatting issues because Claude doesn't know my oxformat config. The model handles the logic; the hook handles the last 10%.

**Stop** -- I've lost count of how many times Claude stopped mid-task and I didn't notice for twenty minutes because I was running five sessions in tmux. A Stop hook can nudge Claude to keep going, or spin up a verification agent to check the work before declaring victory.

**PostCompact** -- This one bit me before I knew it existed. Claude compresses its context window during long sessions, and instructions can get silently dropped. I had a session where Claude "forgot" my project's naming conventions halfway through a refactor. PostCompact re-injects whatever you can't afford to lose. If I'd had this hook a month ago, I'd have saved an afternoon of cleanup.

## From One Hook to Five

After reading Boris's thread and digging into the docs, here's what I'm adding. Not all at once -- I'll validate each one before moving on.

### Auto-Format on Write

The most obvious win. My blog is a Nuxt project. Every time Claude writes or edits a file, run the formatter. I use [oxfmt](https://oxc.rs/docs/guide/usage/formatter) (from the oxc toolchain) instead of Prettier — it's faster and already part of my lint setup. And since `oxfmt` already knows which file types it can format, the hook stays simple:

```json
"PostToolUse": [
  {
    "matcher": "Write|Edit",
    "hooks": [
      {
        "type": "command",
        "command": "bash -c 'INPUT=$(cat); FILE=$(echo \"$INPUT\" | jq -r \".tool_input.file_path // empty\"); if [ -n \"$FILE\" ]; then npx oxfmt --write \"$FILE\" 2>/dev/null; fi; exit 0'"
      }
    ]
  }
]
```

No more formatting drift. No more "Claude wrote valid code but the linter hates the output" commits. The hook handles it silently every time.

### The Force-Push That Almost Was

I mentioned that force-push scare. Permissions caught it, but I spent the next hour wondering what _else_ might slip through. What about `rm -rf /`? What about `DROP TABLE`? Permissions are a gate. This hook is the alarm system:

```json
"PreToolUse": [
  {
    "matcher": "Bash",
    "hooks": [
      {
        "type": "command",
        "command": "bash -c 'INPUT=$(cat); CMD=$(echo \"$INPUT\" | jq -r \".tool_input.command\"); if echo \"$CMD\" | grep -qE \"rm\\s+-rf\\s+/|git\\s+push\\s+--force\\s+.*main|DROP\\s+TABLE\"; then echo \"BLOCKED: $CMD\" >&2; exit 2; fi'"
      }
    ]
  }
]
```

Exit code 2 is the magic number. It tells Claude the action was denied, and the error message on stderr gets sent back as context. Claude doesn't just fail -- it understands _why_ and adjusts.

### Fresh Context, Every Session

This is the one I'm most excited about. Instead of maintaining a massive CLAUDE.md that covers every scenario, or manually pasting context every morning like a caveman:

```json
"SessionStart": [
  {
    "matcher": "",
    "hooks": [
      {
        "type": "command",
        "command": "bash -c 'echo \"Recent changes:\"; git log --oneline -5 2>/dev/null; echo \"\\nOpen issues:\"; gh issue list --limit 3 --json title,number --jq \".[] | \\\"#\\(.number): \\(.title)\\\"\" 2>/dev/null'"
      }
    ]
  }
]
```

Every time Claude starts, it sees the latest git history and open issues. No stale context. No copy-paste rituals.

## The Gotcha That Cost Me an Hour

My first custom hook did nothing. No errors, no output, just silence. I stared at it. I re-read the config. I added `echo "HELLO"` at the top and confirmed it wasn't running. Then I added `echo "HELLO" >&2` and it _was_ running -- just silently succeeding with no visible output.

But the real problem was dumber than that. Hooks receive event data as JSON on **stdin**. Not environment variables. Not arguments. Not a magical `$CLAUDE_TOOL_INPUT` variable that I definitely invented and tried to use.

Stdin. You read it with `cat`. Like it's 1975.

```bash
#!/bin/bash
INPUT=$(cat)
TOOL_NAME=$(echo "$INPUT" | jq -r '.tool_name')
FILE_PATH=$(echo "$INPUT" | jq -r '.tool_input.file_path // empty')
```

Once you know, it's obvious. But nothing in the JSON configuration syntax hints at "by the way, your script needs to read from stdin" -- you either find it in the docs or you stare at a silent script questioning your career choices.

## Exit Codes as a Control Plane

The exit code system is elegant:

- **0** -- success, proceed normally
- **2** -- block the action (PreToolUse only), send stderr to Claude as explanation
- **anything else** -- hook error, Claude continues (fail-open)

That fail-open default is smart. A buggy hook script doesn't brick your session. It logs the error and moves on. You get reliability without fragility.

## Three Scopes, Three Purposes

Hooks live in three places, each with a different audience:

| Location                      | Scope                          | Committed to git? |
| ----------------------------- | ------------------------------ | ----------------- |
| `~/.claude/settings.json`     | All projects on this machine   | No                |
| `.claude/settings.json`       | This project, shared with team | Yes               |
| `.claude/settings.local.json` | This project, just me          | No (gitignored)   |

Safety gates and formatting go in the project-level file so the whole team gets them. Notification and personal preferences stay in my user-level config.

## The Compounding Effect

Each individual hook is small. Auto-format saves maybe 30 seconds per edit. The safety gate prevents a disaster that might never happen. Context injection saves a few copy-pastes per day.

But they compound. Five hooks running across five parallel Claude sessions means 25 automated interventions happening without me thinking about them. That's the real capability I was leaving on the table -- not any single hook, but the aggregate effect of deterministic code running at every lifecycle moment while I focus on something else.

It's the difference between telling a junior dev "always run the linter" and adding a pre-commit hook that does it for them. Both communicate the same intent. Only one guarantees it happens.

I opened my settings file last week and found one hook on an empty wall. Time to hang some more.
