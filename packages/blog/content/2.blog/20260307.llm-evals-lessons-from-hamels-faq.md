I read [Hamel Husain and Shreya Shankar's LLM evals FAQ](https://hamel.dev/blog/posts/evals-faq/) a few weeks ago. It's drawn from teaching 700+ engineers and it's dense with opinions. Not all of it clicked immediately, but a few ideas hit hard because I'd already been making the mistakes they describe.

This isn't a summary of their post. It's what changed in how I think about the [chatbot on this blog](/chat) after reading it.

## Error Analysis Over Infrastructure

The FAQ's core claim: spend 60-80% of your time reviewing failures, not building eval infrastructure. I felt called out.

When I built the chat feature on this blog, I did what most developers do. I wired up [Braintrust](https://www.braintrust.dev/) for observability, wrapped the Anthropic client with their SDK, and felt productive:

```ts
import { wrapAnthropic, initLogger } from 'braintrust';

const rawClient = new Anthropic({ apiKey: result.ANTHROPIC_API_KEY });
getBraintrustLogger();
_client = wrapAnthropic(rawClient);
```

Observability is there. Traces are logging. But when the chatbot gives a bad answer, what do I actually _do_ with those traces? Mostly nothing. I look at the dashboard, nod at the charts, and move on.

Hamel's point is that the boring work -- sitting down and reading 50 bad outputs -- is where the real improvement comes from. Not the dashboard. Not the metrics pipeline. The reading.

I've started doing this manually. Open Braintrust, filter for longer conversations (those tend to surface more issues), and just read. No scoring rubric. Just: "Would I be embarrassed if someone saw this response?" That question alone surfaces more actionable problems than any automated metric I've tried.

## Binary Pass/Fail Over Likert Scales

This one was counterintuitive. My instinct was to score chatbot responses on a 1-5 scale. How helpful was it? How accurate? How well-formatted?

The FAQ argues for binary: pass or fail. The reasoning is that Likert scales create false precision. What's the difference between a 3 and a 4? Nobody agrees, not even the same person on different days.

With the blog chatbot, I've been applying this to the system prompt. The prompt has explicit formatting rules:

```
**FORMATTING RULES (CRITICAL):**
- ABSOLUTELY NO MARKDOWN HEADINGS: Never use #, ##, ###
- Use **bold text** for emphasis and section labels instead
- Start all responses with content, never with a heading
```

Did the response use headings? Pass or fail. Did it cite blog posts with working links when it used the search tool? Pass or fail. Did it answer in a reasonable length? This one's harder to make binary -- but "did it exceed 3 paragraphs for a simple question" works.

Binary forces you to define what "good" actually means. Likert lets you punt on that definition.

## The Benevolent Dictator

Hamel recommends a "benevolent dictator" model for annotation: one domain expert decides what quality means, not a committee. Consensus annotation sounds democratic. In practice, it produces averaged-out, watered-down quality standards.

For a personal blog chatbot, this is easy -- I'm the only one who cares. But the principle scales. When I built the [multi-agent loan approval demo](/blog/building-a-multi-agent-loan-approval-system-with-human-in-the-loop), each AI reviewer has a distinct personality and evaluation criteria defined in its own skill file. The Bank is conservative. Background Checks is paranoid. They don't try to agree -- they each apply their own lens.

That's the dictator model applied to AI reviewers. One perspective per reviewer, no averaging. The human at the end synthesizes, but each reviewer is uncompromising about their own criteria.

## Build Custom Annotation Tools

The FAQ is skeptical of off-the-shelf annotation platforms. Their recommendation: spend a few hours with AI coding tools and build something custom that fits your exact workflow.

I accidentally did this. The blog's chat system stores every conversation in PostgreSQL with structured message parts -- text, reasoning, code executions, tool calls, file references. Each message is broken into typed parts:

```ts
// Every response is stored with its full structure
const messageParts: MessagePart[] = [
  ...(reasoningText ? [{ type: 'reasoning', text: reasoningText, state: 'done' }] : []),
  ...codeExecutions,
  ...fileParts,
  ...(fullText ? [{ type: 'text', text: fullText }] : []),
];

await db.insert(tables.messages).values({
  chatId: chat.id,
  role: 'assistant',
  parts: messageParts,
});
```

I built this for the product, not for evals. But it turns out that structured storage _is_ your annotation tool. I can query for conversations where the model used code execution, or where it called the blog search tool but the user asked a follow-up (suggesting the search result wasn't helpful), or where responses included reasoning tokens above a certain length (suggesting the model struggled).

The custom tooling isn't a separate "eval platform." It's the same database, queried differently.

## Skepticism of Automation

This is the one I'm still wrestling with. The FAQ warns against automated eval pipelines that prevent you from building intuition. If you automate too early, you optimize for metrics without understanding the failure modes.

I have a `searchBlogContent` tool that does RAG retrieval -- hybrid search with pgvector cosine similarity and BM25 full-text, fused with Reciprocal Rank Fusion. It sounds sophisticated. But when a user asks "what does Chris think about testing?" and the tool returns chunks about Vitest configuration instead of the [philosophical post about why tests matter](/blog/you-do-not-have-time-to-not-have-tests), the sophistication doesn't matter. I only discovered this failure pattern by reading conversations, not by watching retrieval metrics.

The temptation is to build an automated eval that checks retrieval relevance with an LLM judge. The FAQ's advice: don't, not yet. First, build a taxonomy of failure modes from manual review. _Then_ automate the ones you understand well enough to write a binary pass/fail check for.

I'm not there yet. I'm still in the "read conversations and take notes" phase. And that's apparently where I should be.

## What Actually Changed

After reading the FAQ, I made three concrete changes:

1. **Weekly review sessions.** I spend 30 minutes reading chatbot conversations in Braintrust. No scoring, just reading with the question "would I be embarrassed?"

2. **Binary checks in the system prompt tests.** The existing unit tests for the chat endpoint already validate SSE format and message parsing. I'm extending them with binary assertions about the system prompt's constraints -- does it prohibit headings, does it require conciseness, does it include tool descriptions. These aren't eval-driven development; they're guardrails on the prompt itself.

3. **A failure log.** A plain text file where I note patterns. "Search tool returns config-level content when user wants opinion-level content" is one entry. "Model apologizes excessively when it doesn't know something" is another. These become the spec for future automated evals -- but only once I've seen the pattern enough times to define pass/fail precisely.

None of this is glamorous. That's the point. The FAQ's thesis is that the unglamorous work -- reading outputs, making binary judgments, building intuition -- is the work that actually improves your system. Everything else is infrastructure cosplay.
