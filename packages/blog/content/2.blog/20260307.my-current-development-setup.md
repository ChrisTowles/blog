This is a snapshot of how I work right now — the tools, the workflow, and how AI fits into it. It changes fast, so I wanted to capture where things stand in March 2026.

## The AI Work Loop

Claude Code CLI is the center of my workflow. I run it in the terminal, not through an IDE plugin. It reads my codebase, runs commands, edits files, and commits — all with permission checks at each step.

### CLAUDE.md Instructions

Every project has a `CLAUDE.md` at the root with project conventions, commands, and gotchas. Claude reads this automatically. I also have a global `~/.claude/CLAUDE.md` with cross-project preferences like "always rebase merge" and "use ES modules." The blog project has nested `CLAUDE.md` files in subdirectories (`packages/blog/CLAUDE.md`, `packages/blog/server/CLAUDE.md`) for context-specific rules.

### Skills

Skills are reusable prompt templates stored in `.claude/skills/`. I have 11 right now — things like `blog-01-create-prompt` through `blog-04-image-prompt` for the blog content pipeline, `nuxt-v4` for framework conventions, `slidev` for presentations, and domain-specific ones for different features. They give Claude deep context without burning tokens repeating instructions.

### Agents

I have a `blog-content-architect` agent that enforces tone and voice. It maintains the authentic, concise, experience-driven style — personal learning narratives rather than authority positioning. Expert-to-expert, not teacher-to-student. It is modeled after Matt Pocock's writing style.

### The Ralph Loop

[Ralph](/blog/implementing-ralph-wiggum-loop-for-autonomous-ai-coding) is my autonomous coding loop (`scripts/ralph-loop.ts`). You feed it tasks, and it runs Claude CLI iterations until they are done. Each iteration picks a task, works on it, and reports back. It is useful for grinding through multi-step work like "add tests for these 5 modules" without babysitting each one.

### Claude on GitHub

The `claude.yml` workflow triggers Claude Code via `@claude` mentions in issues and PRs. It uses `anthropics/claude-code-action@beta` — so I can assign work to Claude directly from GitHub issues.

## Project Structure

This blog is a pnpm monorepo:

```
packages/
├── blog/           # Nuxt 4 + Nuxt UI application
│   ├── app/        # Client: components, composables, pages
│   ├── server/     # Nitro: API routes, database, AI integrations
│   └── shared/     # Types shared between client and server
├── slides/         # Slidev presentations
infra/
├── container/      # Dockerfile for Cloud Run
├── terraform/      # GCP infrastructure (Cloud Run, Cloud SQL)
└── aws_cloudformation/  # AWS Bedrock IAM for embeddings
```

The `packages/blog` directory is where almost everything lives. Nuxt 4's folder layout splits client (`app/`) from server (`server/`), with `shared/` for types that cross the boundary.

## Dev Tools

**Linting**: oxlint — fast, no config files, replaces ESLint. Formatting with oxfmt. Both run in pre-commit hooks via `simple-git-hooks` and `lint-staged`.

**Testing**: Vitest for unit and integration tests, Playwright for E2E. Integration tests hit a real PostgreSQL (pgvector) running in Docker. Test files sit next to source (`foo.ts` / `foo.test.ts`), integration tests use `.integration.test.ts` suffix with a separate Vitest config.

**Database**: Drizzle ORM with PostgreSQL. Schema in TypeScript, migrations generated with `drizzle-kit`. pgvector for embeddings, custom SQL migrations for tsvector columns and GIN indexes that Drizzle can't express natively.

**TypeScript**: Strict mode, `pnpm typecheck` runs `nuxt typecheck`. Zod for runtime validation on every API route.

**Git worktrees**: `scripts/worktree.ts` manages isolated worktrees with slot-based port allocation. Each slot gets pre-configured ports and OAuth apps so multiple branches can run dev servers simultaneously without conflicts. This is what lets me have Claude working on one branch while I work on another.

## Terminal Setup

Zsh with Oh My Zsh. I have written about [setting up Zsh on servers](/blog/oh-my-zsh-on-your-server) before. Key tools beyond the basics:

- **pnpm** — package manager, workspace protocol for the monorepo
- **tsx** — runs TypeScript scripts directly (build scripts, worktree manager, Ralph loop)
- **zx** — Google's tool for shell scripts in JS/TS, used heavily in build and deploy scripts
- **Docker Compose** — local PostgreSQL with pgvector for development

## Deployment

**GCP Cloud Run** hosts everything. A single container runs the Nuxt server (SSR + API).

**Terraform** manages the infrastructure — Cloud Run service, Cloud SQL (PostgreSQL), secrets, IAM. Two environments: `staging` and `prod`, each with their own `.tfvars` and backend config. The deploy script (`scripts/build.ts`) builds the container, pushes to Artifact Registry, and updates the Cloud Run revision.

**CI/CD** is two GitHub Actions workflows:

- `test.yml` — lint, typecheck, unit tests, integration tests (with a pgvector service container)
- `deploy.yml` — runs tests, builds the container image, deploys to both staging and prod on every push to `main`

**Cloudflare** handles DNS only. I migrated away from Cloudflare Workers hosting to Cloud Run for the PostgreSQL access and container flexibility.

## Blog AI Features

The blog itself is an AI playground with three main features:

### Chat (`/chat`)

Conversational AI with tool use. The server streams responses via SSE using Anthropic's beta APIs. Tools include blog search (RAG-powered), weather, dice rolling, and document generation (PDF/PPTX/XLSX/DOCX via the Skills API). Code execution runs in Anthropic's sandboxed containers.

### RAG Search

Blog content gets chunked, summarized with Claude Haiku (with prompt caching), and embedded with AWS Bedrock Titan Text v2. Search uses hybrid retrieval — pgvector cosine similarity plus BM25 full-text search — fused with Reciprocal Rank Fusion. Optional Cohere v3 reranking on top.

### Artifacts

Interactive code execution embedded in blog posts via the `::code-runner` MDC component. Code runs in Anthropic's isolated containers (no internet access), results stream back via SSE. Generated files (charts, documents) are fetched through the Anthropic Files API.

## What Changes Fast

The AI tooling layer evolves weekly. Skills get added and refined. The Ralph loop gets smarter. New Anthropic beta features land and I wire them in. The infrastructure and dev tools are more stable — Terraform, Drizzle, and the testing stack have been solid for months.

If you are setting up something similar, the biggest leverage comes from `CLAUDE.md` files and git worktrees. The `CLAUDE.md` files compound — every convention you document pays off across every future session. And worktrees let you parallelize work across multiple Claude instances without branch conflicts.
